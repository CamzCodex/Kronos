# Experiment and model registry

Status: implemented contract; no real experiment is registered by this document  
Registry version: `1.0.0`  
Operating mode: research and paper simulation only

## Purpose

`kronos_eval.registry` turns an evaluation result and its supporting files into a
reconstructable local record. It closes the gap between a caller declaring an artifact hash
and the repository proving that bytes with that hash were actually copied and can still be
read.

The registry is deliberately local and dependency-light. It does not introduce MLflow, a
hosted service, a broker connection, or a model-serving path.

## Required experiment lineage

Every registration contains:

- full Git commit and dirty-tree declaration;
- dataset ID;
- complete canonical configuration and its independently checked SHA-256 identity;
- feature-schema version;
- model identifier and checkpoint revision;
- seed and fold definitions;
- hardware and library versions;
- metrics;
- one or more byte-verified artifacts;
- initial approval status and any required evidence references;
- creation time and optional superseded experiment ID.

Configurations, folds, metrics, hardware, and version metadata must be finite canonical JSON.
An empty artifact set, empty folds, digest mismatch, duplicate artifact role, naive timestamp,
or dangling supersession reference is refused.

## Filesystem layout

```text
registry-root/
  artifacts/sha256/<prefix>/<sha256>
  experiments/<experiment-id>.json
  aliases/<alias>.json
  alias-events/<alias>/<event-id>.json
```

Artifacts are streamed to a same-filesystem temporary file while their SHA-256 digest is
calculated. The temporary file is linked into its content-addressed destination only after the
declared digest matches. Experiment records and alias events are immutable. An alias pointer is
replaceable, but every replacement must have a strictly later timestamp and is backed by an
immutable, hash-identified history event.

The registry detects a missing or modified artifact during reconstruction. It also recomputes
the experiment identity, configuration hash, alias-event identity, and alias-history chain.

## Status and alias policy

Initial approval status is one of:

- `unreviewed`;
- `approved`;
- `research-only`;
- `rejected`.

An `approved` or `rejected` registration requires at least one approval-evidence reference.
That reference is identity-bound but is not authenticated or fetched by registry version 1.

Supported aliases are `candidate`, `champion`, `rollback`, `research-only`, and `rejected`.
`champion` and `rollback` require an approved, clean-tree, artifact-verified record. A rejected
record cannot become `candidate`; the remaining status-specific aliases enforce matching
status policy.

This policy is a mechanical eligibility gate only. It does not establish that an approval was
scientifically justified, that a dataset was causal, or that Kronos has economic value.

## Reconstruction contract

`ExperimentRegistry.get(experiment_id)` succeeds only when:

1. the immutable record exists and parses;
2. its canonical identity recreates the requested experiment ID;
3. configuration and decision-state fields are internally consistent;
4. every content-addressed artifact exists at the expected path;
5. every stored byte count and SHA-256 digest matches.

Deleting an original source file does not break reconstruction because the registry retains the
verified content-addressed copy. Changing or deleting the registered copy invalidates the run.

## Known limitations

- Git commit, dirty-tree state, hardware, library versions, approval references, and source
  semantics are declared by the caller rather than independently attested.
- Local files are tamper-evident when read, not signed, remotely replicated, or immutable
  against an administrator.
- Alias updates do not yet use a cross-process lock; concurrent writers must be serialized by
  the experiment launcher.
- Registry identity proves exact bytes and metadata, not that upstream data were licensed,
  point-in-time, survivorship-safe, or causally prepared.
- MLflow-compatible export is deferred until local evidence proves that the additional
  dependency and service surface are justified.

No synthetic fixture generated by the registry tests is an experiment result and none is added
to `research/EXPERIMENT_REGISTER.jsonl`.
